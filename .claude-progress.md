## PART 1 — SQLite-Specific Fixes in Canonical Path

### Issue Found
In `backend/app/routers/crypto_payments.py` function `_record_invoice_crypto_payment()`:

1. **Line 105**: `lower(hex(randomblob(8)))` — SQLite-specific ID generation
   - **Why SQLite-specific**: `randomblob(N)` is a SQLite-only function that returns N random bytes
   - **Fix**: Replaced with Python `str(uuid.uuid4())` — database-agnostic UUID generation

2. **Line 141**: `datetime('now')` — SQLite-specific timestamp
   - **Why SQLite-specific**: `datetime('now')` is SQLite date/time function syntax
   - **Fix**: Replaced with Python `datetime.now().isoformat()` captured before query execution

3. **Line 150**: `datetime('now')` — Same SQLite-specific timestamp issue

### Code Changed
```python
# Before (SQLite-specific):
VALUES (lower(hex(randomblob(8))), ?, ?, ?, ?, ?, ?, ?)
updated_at = datetime('now') WHERE id = ?

# After (DB-agnostic):
VALUES (?, ?, ?, ?, ?, ?, ?, ?)  # using str(uuid.uuid4())
updated_at = ? WHERE id = ?     # using now_str captured earlier
```

## PART 3 — PaymentModule Canonical Migration

### Old Call Chain
```
PaymentModule loads → useEffect fires
→ fetch('/crypto-checkout/addresses')
→ Returns static wallet addresses from env vars
→ Display hardcoded address + 30-min countdown
→ No actual payment status tracking
→ No invoice linkage
```

### New Call Chain (Canonical Path)
```
PaymentModule loads → useEffect fires
→ If orderId provided:
  → fetch('/crypto-payments/invoice/{orderId}/generate-address?token=USDC')
  → Returns canonical payment: address, amount, expiry, status
  → Start polling '/crypto-payments/invoice/{orderId}/status' every 15s
  → Display actual status: pending → confirming → confirmed/expired
  → Countdown based on actual expires_at
→ If canonical fails OR no orderId:
  → Fallback to legacy '/crypto-checkout/addresses'
  → Display static addresses with 30-min countdown
```

### UI States Added/Changed

| State | Old Behavior | New Behavior |
|---|---|---|
| Loading | "Loading crypto options..." | Same |
| No wallets | "No crypto wallets configured" | Same |
| Address display | Static from env | From canonical API when available |
| Timer | 30 min hardcoded | From `expires_at` in canonical response |
| Payment status | Always "Waiting for payment..." | Honest: pending/confirming/confirmed/expired |
| Token switching | No re-fetch | Re-calls generate-address with new token |
| Legacy badge | None | Shows "Legacy mode" tag in fallback |

### Fallback Rules
1. If `orderId` is provided, try canonical first
2. If canonical `/generate-address` fails, fall back to legacy `/crypto-checkout/addresses`
3. If canonical `/status` poll fails, continue showing last known status
4. Legacy mode shows "Legacy mode" badge so users know it's not canonical
