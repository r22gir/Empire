## Prior Session Summary

### EmpirePay Architecture (Two Parallel Systems Found)

**System A — `crypto_payments.py` (canonical):**
- Uses SQLAlchemy models (`CryptoPayment`, `CryptoLedger`) with PostgreSQL
- Full payment lifecycle: create → confirm → ledger entries
- Supports BTC, ETH, SOL, USDC, USDT, BNB, BUSD, ADA, EMPIRE across bitcoin/ethereum/solana/bnb/cardano chains
- Discounts: 15% crypto, 20% EMPIRE token
- Wallet addresses derived deterministically via HMAC-SHA256 from `CRYPTO_MASTER_SEED`
- Has explicit failure: `_require_crypto()` raises HTTPException 503 if seed not set
- **NOT connected to finance DB** — mirrors payment record logic manually
- Endpoints registered at `/api/v1/crypto-payments` (tag: `crypto`)

**System B — `crypto_checkout.py` (legacy lightweight):**
- Uses SQLite finance DB directly
- Static wallet addresses from `.env` vars
- Only BTC/ETH/SOL/USDT with 10% discount (no EMPIRE)
- CoinGecko price fetching with 60-second cache
- Customer notifies → owner verifies manually
- **NOT using the SQLAlchemy models**
- Endpoints registered at `/api/v1/crypto-checkout` (tag: `crypto-checkout`)

**System C — `payments.py` (Stripe, separate concern):**
- SaaS subscriptions + invoice payment links via Stripe
- **Not EmpirePay** — separate Stripe-based payment system
